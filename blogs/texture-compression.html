GPUs interpolate textures. BC1 explained.

I was reading a paper on neural texture compression when I hit a footnote that sent me down a rabbit hole for three hours. The footnote mentioned BC1. I thought I knew what it was. I didn't.

Every texture in every game you've played, the brick wall, the rust, the grass, was never stored as real colors. Your GPU has been reconstructing approximate colors from a handful of numbers, billions of times per frame, and it's been doing this since 1998.

The Core Problem

A single 4K texture in raw RGBA is 32MB. A modern game has thousands of them. VRAM is fast, but it's not infinite, and it has a cache. If your texture data doesn't fit in the cache you pay a penalty every sample. Every frame. For every visible surface.

The obvious answer is compression. But you can't just zip a texture. The GPU needs to decompress individual pixels on the fly, in hardware, in the middle of rendering. The decompressor has to be trivially cheap: no entropy coding, no variable-length anything, just a formula the hardware can run in one clock.

What BC1 Is

S3 Graphics invented it in 1998 and called it DXT1. Microsoft licensed it into DirectX 6. It was renamed BC1 in DirectX 10. Every DirectX-compliant GPU has decoded it in hardware ever since. Same bits, same format, three different names depending on the era.

The 4×4 Block and the Budget

BC1 works on 4×4 blocks of pixels. Each block is 16 pixels. Raw, that's 16 × 4 bytes = 64 bytes. The target is 8× compression: 8 bytes per block. That's the only goal. Everything else falls out of that budget.

Those 8 bytes split into two parts:

8 bytes total
├── 4 bytes  →  2 color endpoints (C0, C1) in RGB565
└── 4 bytes  →  16 pixels × 2 bits each = 32 bits

Spending 4 bytes on two color endpoints was the design decision. The remaining 4 bytes forced everything else.

Why Endpoints

The alternative is to give each pixel 4 bits and quantize to a global 16-step palette. The problem: if your block is a dark patch, say all pixels between values 40 and 80, maybe two of those 16 global steps land in your range. Fourteen steps wasted. Big jumps. Visible banding.

Endpoints fix this. You spend 4 bytes to describe the block's actual color range, then subdivide just that range into 4 steps. All steps useful, all steps tight. Adaptive precision per block instead of fixed global precision.

It's the same idea as floating point vs fixed point: spend some bits describing the range, then use the remaining bits to subdivide it. More precision ends up where it matters.

The Line in RGB Space

C0 and C1 are not scalars. They're vec3, full RGB colors. Together they define a line segment in 3D RGB space. Moving along that line with a single scalar t changes R, G, B proportionally:

lerp(C0, C1, t) = (
    R0 + t*(R1-R0),
    G0 + t*(G1-G0),
    B0 + t*(B1-B0)
)

One scalar drives all three channels. That's why two endpoints are enough to describe an entire color range.

The 4 Colors

2 bits per pixel means 4 possible values. Those map to 4 equally spaced t values along the line:

00  →  t = 0    →  C0                 (stored)
01  →  t = 1/3  →  lerp(C0, C1, 1/3)  (derived)
10  →  t = 2/3  →  lerp(C0, C1, 2/3)  (derived)
11  →  t = 1    →  C1                 (stored)

Only C0 and C1 are stored. The middle two are always recomputed by the GPU. They're perfectly predictable from the endpoints. Every pixel in the block must be one of these four colors. That constraint is the compression.

What the Index Actually Is

The 2-bit index is a quantized scalar t. Each pixel has its own independent t, but you can't store 16 floats (512 bits). So each t is rounded to the nearest of the 4 allowed values and stored as 2 bits:

exact t = 0.72  →  nearest = 2/3  →  store 10
exact t = 0.11  →  nearest = 0    →  store 00
exact t = 0.94  →  nearest = 1    →  store 11

The rounding error is the lossy part. That's where color information is permanently lost.

Why You Need Both

Endpoints and indices do different things:

C0, C1   →  what 4 colors are possible in this block
indices  →  which of those 4 colors each pixel actually is

Endpoints without indices: you know the palette but not the painting. Indices without endpoints: you know the arrangement but not what colors to put where. Both are necessary. Neither is redundant.

Finding the Best Line

Given 16 pixels as points in 3D RGB space, finding the best C0 and C1 is a line-fitting problem. That's PCA.

Compute the mean color of the 16 pixels, build a covariance matrix, find its largest eigenvector (the axis of greatest color variance), project all pixels onto that axis, and take the min and max projections as C0 and C1. Each pixel then snaps to the nearest of the four colors and stores its 2-bit index.

PCA doesn't change any pixel's color. It just finds the best line. The color loss happens when each pixel's exact t gets rounded to the nearest of 4 values.

When Does This Happen

Offline (build pipeline)
  PNG → texconv → PCA runs, endpoints found, indices assigned
  output: .dds file (BC1 bytes on disk)

Load time
  .dds → CreateTexture2D() → VRAM
  no decompression, just a memcpy

Runtime (every frame, every sample)
  myTex.Sample(sampler, uv)
  → GPU fetches 8-byte block from VRAM
  → reads 2-bit index for this pixel
  → lerps between C0 and C1
  → returns color to shader
  one clock, transparent to your HLSL

The expensive part never touches your game. At runtime the decoder is a shift, a mask, and a lerp.

Why BC1 Is Faster, Not Just Smaller

A 256×256 BC1 texture is 32KB. Raw RGBA is 256KB. Eight times more texture fits in the GPU's L2 cache at once. Cache hit rate goes up, memory bandwidth pressure goes down. You get better performance even before counting the VRAM savings.

Using BC1 in DirectX 11

Compress your texture offline with texconv (part of DirectXTex):

texconv -f BC1_UNORM albedo.png -o textures/

This writes a .dds file with the BC1 data already packed. On the D3D11 side, one field changes:

desc.Format = DXGI_FORMAT_BC1_UNORM;  // instead of DXGI_FORMAT_R8G8B8A8_UNORM

The rest of your pipeline stays the same. The shader sees a normal texture. The hardware handles decompression transparently at sample time.

If you need alpha, swap to DXGI_FORMAT_BC3_UNORM. For higher quality, DXGI_FORMAT_BC7_UNORM is the DX11 option, though the encoder is slower offline.

Where It Breaks Down

BC1 assumes all 16 pixels in a block cluster along a single line in RGB space. When that holds (smooth gradients, solid surfaces, highlights) quality is excellent. When it breaks (hard edges between two different colors bisecting a block, complex multi-color patterns) you get visible block artifacts. The block can only have 4 colors. If the real content needs more, it shows.

Where It Goes Next

BC7 (DirectX 11) splits blocks into sub-regions, each with their own endpoint pair. Multiple PCA fits per block. Better quality, double the byte cost.

In 2024, AMD asked what happens if you replace the line-finding step entirely with a small neural network. Instead of assuming colors fit a line, a tiny network learns the best possible mapping for each block. Same bit budget, better quality. That's NTBC, the paper that started this rabbit hole. Understanding BC1 (the PCA, the lerp, the 2-bit index, the quantization step) is what makes that paper land.

The best compression schemes are the ones nobody thinks about. BC1 has been running in every GPU for 26 years. That's good engineering.
